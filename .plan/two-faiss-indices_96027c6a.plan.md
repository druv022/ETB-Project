## Goal

Produce two vector stores from one run:

- `text_vectorstore`: embeddings for text chunks
- `caption_vectorstore`: embeddings for image captions (as separate documents)

## Key design

- Keep captioning modular via existing `ImageCaptioner` interface (already supports OpenRouter).
- Preserve backward compatibility: keep `process_pdf()` returning only text chunk `list[Document]`.
- Build both FAISS stores in a single end-to-end dual-index pipeline function (programmatic + CLI).

## Implementation plan

### 1. Preserve `process_pdf()` API; add internal helper + dual-index entrypoint

- Update `[src/etb_project/document_processing/processor.py]`:
  - Keep `process_pdf(...) -> list[Document]` unchanged (still returns text chunks only).
  - Introduce an internal helper that returns `(text_chunks, caption_docs)`.
  - Expose a new higher-level function that the dual-index pipeline can call to get both collections.

### 2. Generate caption documents alongside text chunks

- In the internal helper (after captioning but before returning):
  - Create `caption_docs: list[Document]` from `images_by_page` and generated captions.
  - For each extracted image with non-empty caption:
    - `page_content` should be prefixed because you selected `caption-prefixed`:
      - `page_content = "Image caption: {caption}"`
    - `metadata` should include (at minimum) fields to trace back to the source image:
      - `source`, `page` (1-based), `image_index`, `xref`, `path`, `ext`
      - optionally `total_pages` and `caption_source`.
  - Return `(chunk_docs, caption_docs)`.
  - The public `process_pdf()` wrapper should discard `caption_docs` and return only `chunk_docs`.

### 3. Keep `pages.json` serialization compatible

- Existing serialization already writes captions into `pages.json` images entries.
- Ensure caption generation uses the same caption list used for `pages.json`, so caption docs and serialized captions stay consistent.

### 4. Build two FAISS vector stores in one place

- In `[src/etb_project/retrieval/process.py]` add a function, e.g.:
  - `build_two_vectorstores(text_docs: list[Document], caption_docs: list[Document]) -> tuple[FAISS, FAISS]`
  - Internally call existing `process_prechunked_documents()` twice.
- Add an end-to-end convenience function (optional but recommended) to keep the pipeline “single”:
  - `process_pdf_to_vectorstores(..., image_captioner=...) -> (text_vectorstore, caption_vectorstore)`

### 5. Update the standalone document processor CLI

- In `[src/etb_project/document_processor_cli.py]`:
  - When `--build-faiss` is set:
    - build both vector stores via the new dual-index pipeline entrypoint
    - log counts for both stores
  - Preserve existing CLI behavior when a captioner is not configured (caption store may be empty/not built).

### 6. Testing

- Keep existing `[tests/test_document_processing.py]` coverage for `process_pdf()` unchanged (API preserved).
- Add new tests for:
  - caption-doc generation via the dual-index pipeline entrypoint
  - two-index building in retrieval code (mock embedding/FAISS so tests remain fast and deterministic).
- Add unit tests around FAISS building without requiring embeddings:
  - Mock `process_prechunked_documents` in retrieval code so tests only validate “two calls made with correct doc sets”.

### 7. Documentation

- Update `[README.md]` standalone processor section:
  - Explain that the pipeline now produces two vector stores (text vs caption).
  - Describe how captions are turned into caption documents.
  - Note that retrieval can be extended later to query both stores.

## Dataflow (conceptual)

```mermaid
flowchart TD
  A[extract_page_documents] --> B[text chunks]
  C[extract_images] --> D[caption_image via ImageCaptioner(OpenRouter)] --> E[caption_docs]
  B --> F[embed+FAISS -> text_vectorstore]
  E --> G[embed+FAISS -> caption_vectorstore]
  F --> H[return/store]
  G --> H
```
